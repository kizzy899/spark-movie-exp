 # 项目规定
 
 ## 1. 文档管理
 
 1.1 所有文字文档（包括 .docx、.md、.txt、.pdf、.xlsx 等文档类文件）必须放置在 `docs/` 目录下，不得出现在项目根目录或其他代码目录中。
 
 1.2 `docs/` 目录结构约定：
 ```
 docs/
 ├── plans/          # 项目计划、任务分解、考核文档
 ├── reports/        # 课程总报告、总结报告
 ├── references/     # 参考资料、附件文档
 └── README.md       # 文档目录索引
 ```
 
 1.3 文档命名规范：
 - 文件名应清晰反映内容，避免使用无意义文件名（如 `新建文档.docx`）
 - 同一类型的文档建议统一命名风格
 - 版本可通过文件夹或文件名中的日期区分
 
 ## 2. 项目管理
 
 2.1 项目计划文档统一放在 `docs/plans/` 目录，包括：
 - 考核目标文档
 - 任务分解与进度跟踪
 - 项目时间线/甘特图
 - 分工安排
 
 2.2 课程总报告和结项文档放在 `docs/reports/` 目录。
 
 2.3 参考资料、数据说明、外部文档放在 `docs/references/` 目录。
 
 ## 3. 代码管理
 
 3.1 源代码文件（.sol、.py、.js、.ts、.java、.rs、.go 等）不得放入 `docs/` 目录。
 
 3.2 `node_modules/`、`.venv/`、`__pycache__/`、构建产物等不应提交到版本控制。
 
 3.3 每个子项目应有其独立的首页 README.md，说明项目用途和运行方式。
 
 ## 4. 版本控制
 
 4.1 提交信息建议采用简洁的中文或英文，说明本次变更内容。
 
 4.2 功能性变更建议分次提交，避免将不相关的内容混在同一个提交中。
 
 4.3 数据文件（如 .rar 压缩包）如体积较大，应确认是否纳入版本控制，必要时可改用 Git LFS。
 
 ## 5. 工作流
 
 5.1 所有文档类修改应先在 `docs/` 对应子目录中进行。
 
 5.2 在添加任何文档前，确认其放置位置符合本规定。
 
 5.3 本规定文件（agent.md）始终位于项目根目录，不随文档目录结构调整。
