# Spark 电影分析项目三项任务总体测试报告

## 1. 测试目标

本次总体测试覆盖项目的三个核心任务：任务一 Spark RDD 电影 Top20、任务二 Spark SQL 性别标签统计、任务三 Spark Streaming 实时评分。测试目标不是平均覆盖全部代码，而是优先验证最可能造成结果失真的数据转换、输出契约和状态累计逻辑，同时检查统一 Web 入口能否正常提供页面与接口。

## 2. 测试方法与环境

### 2.1 自动化单元测试

- 测试框架：Python `unittest`。
- 统一入口：`python tests/run_all.py`。
- 隔离方法：在导入业务模块前 mock `pyspark`、`pyspark.streaming` 和 `pymysql`，把纯 Python 计算逻辑从 Spark 集群与 MySQL 环境中分离出来。
- 断言方式：使用小规模、结果可手算的输入，直接核对解析结果、排序结果、数组结构和累计状态。

这种方式执行快、结果稳定，适合高频回归；它验证的是核心逻辑，不等同于真实 Spark 集群、JDBC 和 MySQL 的完整联调。

### 2.2 Web/API 轻量集成测试

使用 Flask `test_client` 请求统一入口的 4 个页面和 3 个接口：

```text
页面：/  /task1  /task2  /task3
接口：/api/top20  /api/gender-tags  /api/latest
```

测试重点是路由可达、响应类型正确、已有 JSON 结果可被接口读取，以及外部数据库不可用时接口是否能够返回结构化降级结果。

## 3. 三项任务的重点测试

### 3.1 任务一：RDD Top20

任务一重点测试 CSV 解析与输出路径契约：

- 电影标题含逗号且被双引号包裹时，仍能完整解析，例如 `Shawshank Redemption, The (1994)`。
- `MOVIE_DATA_DIR` 环境变量能够覆盖默认数据路径，避免程序只能在开发者电脑上运行。
- Top20 结果写入路径与 Web API 的读取路径一致，防止 Spark 已生成结果但页面找不到文件。

之所以优先测试这些点，是因为 CSV 错列和路径漂移不会一定触发异常，却会悄悄改变电影名称、关联关系或页面数据，是任务一最隐蔽的失败方式。

### 3.2 任务二：Spark SQL 性别标签

任务二是数据清洗和结果对齐最密集的模块，因此覆盖最细：

- `genres` 按 `|` 拆分，覆盖单标签、空字符串、首尾空格和 `(No genres listed)` 脏值。
- 男女分别取 Top 8；标签少于 8 个时不伪造数据。
- 合并男女标签并集，并对一方缺失的标签补 0，保证 `tags`、`male`、`female` 三个数组同下标对齐。
- 空数据返回空数组。
- 校验 JSON 必需字段及 `code/status/message` 契约。
- 校验 SQL dump 中的回车脏字符、重复标签归一化，以及 `Users.dat`、评分 CSV 的解析。

这些测试直接对应柱状图是否会错位、重复计数或展示伪数据。相比只检查“程序能运行”，它们更能证明统计结果可解释、前后端映射可靠。

### 3.3 任务三：Spark Streaming

任务三重点测试跨批次累计状态，而不只测试单批平均值：

- 含逗号电影标题的 CSV 解析。
- Top 记录保留 `rating_count`，避免页面只看到均分却丢失样本量。
- 新批次与历史状态正确累加评分总和与次数。
- 由累计总和与次数计算平均分，并先执行最小评分人数过滤，再取 Top N。
- 多电影批次能够同时更新已有状态和新增状态。

Streaming 最危险的问题是“每个批次单独看都正确，但跨批次累计错误”。因此测试把 `(rating_sum, count)` 作为不变量，直接验证状态转移，而不是依赖时间和真实目录监听，避免异步环境造成不稳定测试。

## 4. 测试结果

| 层级 | 结果 | 结论 |
| --- | --- | --- |
| 自动化单元测试 | 26/26 通过，耗时约 0.01 秒 | 三项任务的核心纯逻辑回归通过 |
| Web 页面冒烟 | 4/4 返回 HTTP 200 | 统一入口和三个任务页面可访问 |
| API 冒烟 | 3/3 返回 HTTP 200 | Top20 与性别标签返回真实结果；Streaming 返回结构化数据库连接失败信息 |
| 真实外部链路 | 本环境未完成 | 真实 Spark、JDBC、MySQL 写入与 Streaming 文件监听仍需在实验环境联调 |

其中 `/api/latest` 的 HTTP 200 表示降级契约可用，不表示 MySQL 实时数据链路已经成功；当前 MySQL 未启动，接口返回空 `times`、空 `movies` 和明确的连接错误信息。

## 5. 结论与后续建议

当前测试已经较好覆盖三类高风险：输入解析错误、结果数组/JSON 契约错误、Streaming 累计状态错误。26 个自动化用例全部通过，页面与接口也能正常响应，说明核心计算和 Web 读取链路具备稳定的回归基础。

交付前仍建议在完整实验环境补一次端到端联调：启动 MySQL 和 Spark，运行任务一、任务二的真实数据计算；向任务三监听目录连续投递至少两个批次；最后核对数据库记录、三个 API 与 ECharts 页面。这样可以补齐 mock 单元测试无法覆盖的 Spark 执行计划、JDBC 驱动、文件监听时序和数据库事务风险。
